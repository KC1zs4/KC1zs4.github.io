Java安全5(15)_CC链总结

2025-02-11
CTF
Java安全, java反序列化

时隔3个月，重新拾起java安全，回顾总结一波

前置要求

  1. S1: Java安全0_调试与分析
  2. S2: Java安全5系列

CC1_TransformedMap

  • env

    • JDK < 8u71

    • maven: commons-collections:3.1 [https://mvnrepository.com/artifact/commons-collections/commons-collections/3.1]

      • 下载后没有反应需要reload以下maven project

    • 分析: 对于sun.reflect.annotation.AnnotationInvocationHandler的源码并无法直接读到Orcale JDK的源码，SunJDK(OracleJDK)只提供java和javax下的源代码，OpenJDK开源但是并不完全一样，建议先Orcale decompile(IDEA)不行再OpenJDK

  1. 利用到的点: Transformers, AnnotationInvocationHandler

    1. 什么是TransformerMap？Answer: TransformedMap是Map的功能扩展类

  2. CC1_TransformerMap链子核心思想

    1. 入口点: sun.reflect.annotation.AnnotationInvocationHandler.readObject()中MapEntry.setValue()处调用checkSetValue()，直接调用到TransformedMap.transform()
    2. 触发点: transformValue()调用transform，这里的Key和Value都是Transformer的实现，这里并无需使用put()
    3. 调用目标方法: InvokerTransformer
    4. 获取Runtime对象: ConstantTransformer
    5. 链式调用: ChainedTransformer组成的TransformerChain，这里的transform()是比较特殊的，前⼀个回调返回的结果，作为后⼀个回调的参数传⼊，而不是一直在一个对象上调用
    6. 详细内容见Java安全5(15)_CC1_TransformedMap

  3. Gadget Chain: 调用栈顺序

    1. 这里entrySet在设置memberValues为TransformedMap时，对调用到TransformedMap的父类AbstractInputCheckedMapDecorator来

  4. poc

    1. AnnotationInvocationHandler实例化：使用反射与constructor绕过私有限制
    2. Runtime进行序列化: 传递反射的Class类
    3. 在innerMap中先设置key,value而不是在outerMap中设置key,value可以避免在本地触发

CC1_LazyMap

  • env

    • JDK < 8u71

    • maven: commons-collections:3.1 [https://mvnrepository.com/artifact/commons-collections/commons-collections/3.1]

      • 下载后没有反应需要reload以下maven project

    • 分析: 对于sun.reflect.annotation.AnnotationInvocationHandler的源码并无法直接读到Orcale JDK的源码，SunJDK(OracleJDK)只提供java和javax下的源代码，OpenJDK开源但是并不完全一样，建议先Orcale decompile(IDEA)不行再OpenJDK

  1. 利用到的点: 动态代理, LazyMap, Transformers, AnnotationInvocationHandler

    1. 什么是LazyMap？Answer: LazyMap也是CC包中Map的功能扩展类

  2. CC1_LazyMap的核心思想: AnnotationinvocationHandler实际上是动态代理

    1. 入口点: sun.reflect.annotation.AnnotationInvocationHandler中嵌套一个LazyMap的AnnotationInvoctionHandler动态代理，随后调用动态代理中的sun.reflect.annotation.AnnotationInvocationHandler.invoke()调用到get()
    2. 触发点: LazyMap.get()一个没有defined的key会自动触发内部transformer.transform()
    3. (下列和TransformedMap一致)
    4. 调用目标方法: InvokerTransformer
    5. 获取Runtime对象: ConstantTransformer
    6. 链式调用: ChainedTransformer组成的TransformerChain，这里的transform()是比较特殊的，前⼀个回调返回的结果，作为后⼀个回调的参数传⼊，而不是一直在一个对象上调用
    7. 详细内容见Java安全5(4)_CC1_LazyMap

  3. Gadget Chain: 调用栈顺序

  4. poc

    1. 需要两个sun.reflect.annotation.AnnotationInvocationHandler，第一个是传入的反序列化对象(entrySet()时触发invoke())，第二个是反序列化对象的memberValues属性设置动态代理为AnnotationInvocationHandler，第二个Handler内部的memberValues为LazyMap –> 第二个memberValues的LazyMap才是触发点，第一个memberValues只是利用动态代理
    2. 这里外面的AnnotationInvocationHandler也可以是任何会在readObject处调用到动态代理的类，动态代理设置的AnnotationInvocationHandler(内部的)才关键
    3. 最后会报错但是命令还是执行了

CC6_LazyMap

  • env

    • 可高版本JDK

    • maven: commons-collections:3.2.1 [https://mvnrepository.com/artifact/commons-collections/commons-collections/3.2.1] 和 commons-collections:3.1 [https://mvnrepository.com/artifact/commons-collections/commons-collections/3.1]

      • 下载后没有反应需要reload以下maven project

  1. 利用到的点: TiedMapEntry, Transformers, LazyMap

  2. CC6_LazyMap核心思想

    1. 入口点: HashMap的readObject中会有hash()，只要有一个key为TiedMapEntry的就可以作为后续链子的入口点
    2. 触发点: LazyMap.get()一个没有defined的key会自动触发内部transformer.transform()
    3. (下列和TransformedMap一致)
    4. 调用目标方法: InvokerTransformer
    5. 获取Runtime对象: ConstantTransformer
    6. 链式调用: ChainedTransformer组成的TransformerChain，这里的transform()是比较特殊的，前⼀个回调返回的结果，作为后⼀个回调的参数传⼊，而不是一直在一个对象上调用
    7. 详细内容见Java安全5(5)_CC6

  3. Gadget Chain: 调用栈顺序

    1. CC6_LazyMap: 基于CC1来的下半段+TiedMapEntry与hash的结合
    2. 查找技巧，idea中方法gd可以查看全局调用到的部分(向前查找)

  4. poc

    1. 使用反射设置payload是防止在本地执行了命令

CC3_TemplatesImpl

  • env

    • jdk < 8u71

    • maven: commons-collections:3.1 [https://mvnrepository.com/artifact/commons-collections/commons-collections/3.1]

      • 下载后没有反应需要reload以下maven project

  1. 利用到的点: TemplatesImpl, InstantiateTransformer, AnnotationInvocationHandler, TrAXFilter

  2. CC3_TemplatesImpl的核心要点

    1. 入口点: HashMap的readObject中会有hash()，只要有一个key为TiedMapEntry的就可以作为后续链子的入口点
    2. !!!触发点: 其实就是LazyMap的链子在最后ChainTransformer的transform()处用了InstantiateTransformer，并且是通过TrAXFilter类构造函数调用到TemplatesImpl内部重写(这里只是简单包装)的defineClass()来加载Runtime载体恶意类而已
    3. 调用目标方法: TemplatesImpl.defineTransletClasses()和newInstance() <- TemplatesImpl.getTransletInstance() <- TemplatesImpl.newTransformer() <- TrAXFilter(Templates templates)
    4. 获取Runtime对象: 恶意类做载体，在构造函数调用
    5. 链式调用: ChainedTransformer组成的TransformerChain，这里的transform()是比较特殊的，前⼀个回调返回的结果，作为后⼀个回调的参数传⼊，而不是一直在一个对象上调用

  3. Gadget Chain: 调用栈顺序

    1. 基本思路和CC6_LazyMap一致，只是这里的触发点是TemplatesImpl的newTransformer()方法
    2. CC6_LazyMap上半段/CC1_LazyMap+TemplatesImpl与TrAXFilter触发点
    3. CC3多出来的点主要是在于调用处: TemplatesImpl和TrAXFilter都是jdk自带的类，一个sink点

  4. poc

    1. TemplatesImpl 中对加载的字节码是有一定要求的：这个字节码对应的类必须是com.sun.org.apache.xalan.internal.xsltc.runtime.AbstractTranslet的子类
      1. 加载的字节码为一个恶意类，且需要是AbstractTranslet的子类，并且通过对象实例化触发恶意命令 -> 使用构造函数

CC2_For4_PQandTransMap

  • env

    • JDK

      • ⽼的利⽤链在commons-collections4中只需要少量修改就可以使用(CC1,CC3,CC6)，但是内部逻辑其实有了较多的不同了，遵循着Transformers的思想找就好了: TransformedMap和LazyMap

    • maven: commons-collections4 4.0以后4.1之前[https://mvnrepository.com/artifact/org.apache.commons/commons-collections4]

      • cc的新版本包，api发生了变化，但是命名不重复，可以共存在一个项目中

  1. 利用到的点: Transformers(用到了TransformedMap系列的思想), PriorityQueue(java.util), TransformingComparator

  2. CC2_For4_PriorityQueue的核心要点

    1. 入口点: PriorityQueue中的readObject()底层调用了comparaotr

    2. 触发点: PriorityQueue在readObject中使用了siftdown思路，进行比较触发了Transformers系列TransformingComparator的transform方法

      1. 在加载时需要使得PriorityQueue中至少有两个元素并且要进行比较

    3. 调用目标方法: InvokerTransformer

    4. 获取Runtime对象: ConstantTransformer

    5. 链式调用: ChainedTransformer组成的TransformerChain，这里的transform()是比较特殊的，前⼀个回调返回的结果，作为后⼀个回调的参数传⼊，而不是一直在一个对象上调用

  3. Gadget Chain: 调用栈顺序

  4. poc

Summary

  1. 不同链子中的思维妙点
    1. CC1_TransformedMap: 链式构造payload、一直反向推理到readObject做起点
    2. CC1_LazyMap: 利用动态代理连接链子(这里外面的AnnotationInvocationHandler也可以是任何会在readObject处调用到动态代理的类)
    3. CC6_LazyMap: 逐级向上查找、get()到Entry的推理、Map自定义键值
    4. CC3_TemplatesImpl: TemplatesImpl和TrAXFilter都是jdk自带的类(一个sink点，但是LazyMap和ChainedTransformer还是CC链的东西，重点在于Transformer)
  2. 总结出一份图例出来
    1. CC1, CC6: InvokeTransformer(执行恶意代码)
    2. CC3: TemplatesImpl + InstantiateTransformer(加载恶意类)
  3. 大致链子可以分为(按照段的算法)
    • 执行命令方式: TransformedMap, TemplatesImpl
    • 构造链子部分: AnnotationInvocationHandler, PriorityQueue(only CC4), LazyMap+AnnotationInvocationHandler, LazyMap+TiedMapEntry

CC链之死

CC链的利用只能在于3.2.2之前和4.1之前

  1. Apache Commons Collections官⽅在2015年底得知序列化相关的问题后，就在两个分⽀上同时发布了新的版本，4.1和3.2.2

  2. p牛原文

    先看3.2.2，通过diff可以发现，新版代码中增加了⼀个⽅法 FunctorUtils#checkUnsafeSerialization ，⽤于检测反序列化是否安全。如果开发者没有设置全局配置 org.apache.commons.collections.enableUnsafeSerialization=true ，即默认情况下会 抛出异常。 这个检查在常⻅的危险Transformer类 （ InstantiateTransformer 、 InvokerTransformer 、 PrototypeFactory 、 CloneTransforme r 等）的 readObject ⾥进⾏调⽤，所以，当我们反序列化包含这些对象时就会抛出⼀个异常
    再看4.1，修复⽅式⼜不⼀样。4.1⾥，这⼏个危险Transformer类不再实现 Serializable 接⼝，也就 是说，他们⼏个彻底⽆法序列化和反序列化了。更绝
