Java安全6(2)_Tomcat整体架构

2024-12-01
CTF
Java内存马, Java安全

整体速览

  1. 前面说到，tomcat作为服务器处理的是底层的网络连接，而我们的Servlet在上面运行，我们可以注意到Servlet直接使用的是现成的HttpServletRequest和HttpServletResponse，这两个就是网络数据的封装，就是有tomcat处理网络后提供的；总的来说，tomcat有以下的功能
    1. 数据流处理：处理 Socket 连接，负责网络字节流与 Request 和 Response 对象的转化，并最终转化为HttpServletRequest和HttpServletResponse
    2. 加载和管理Servlet，以及具体处理 Request 请求
  2. 什么是service呢？
    1. 一个容器可能对接多个连接器，就好比一个房间有多个门。但是单独的连接器或者容器都不能对外提供服务，需要把它们组装起来才能工作，组装后这个整体叫作 Service 组件
    2. Service 只是在连接器和容器外面多包了一层，把它们组装在一起，通过在 Tomcat 中配置多个 Service，可以实现通过不同的端口号来访问同一台机器上部署的不同应用
  3. 下面我们跟随这幅图进入到连接器中

1. Tomcat连接器

连接器的作用与流程

  1. 总的来说，连接器的作用是统一处理各种网络传输协议与I/O，并最终处理生成可供容器内部Servlet进行处理的ServletRequest对象
    1. 网络通信，Socket这些
    2. 应用层协议解析，不同的流
    3. Tomcat Request/Response与ServletRequest/ServletResponse之间的转换
  2. Servlet的工作流程大致如下
    1. 外部传入数据
      1. 网络连接: 监听网络端口，等待连接请求；接受网络连接请求；读取请求中的字节流
      2. 解析：根据具体应用层协议（HTTP/APR）解析字节流，生成统一的Tomcat Request对象
      3. 转换：将Tomcat Request对象转换成标准的ServletRequest提交给Servlet容器
    2. 内部传出数据
      1. 调用Servlet容器，得到ServletResponse
      2. 将ServletResponse转换成Tomcat Response对象
      3. 将Tomcat Response转成网络字节流进行发送

连接器的内部构成

  1. 上面说到tomcat连接器有三个大的作用，Tomcat的设计者设计了3个组件来实现这三个功能：Endpoint、Processor和Adapter，他们的具体结构如下
    1. Endpoint负责字节流：Endpoint负责提供字节流给Processor处理
    2. Processor负责解析：Processor处理好之后提供Tomcat Request对象给Adapter
    3. Adapter负责转换：Adapter将Tomcat Request对象转成标准的ServletRequest提供给Servlet容器

ProtocolHandler组件

  1. Tomcat中连接器使用ProtocolHandler来处理网络连接和应用层协议，主要包含Endpoint和Processor
  2. Endpoint
    1. 具体职责：Endpoint是通信端点，是具体的Socket接收和发送处理器，是对传输层的抽象，所以Endpoint是用来实现TPC/IP协议的
      1. AbstractEndpoint是Endpoint对应的抽象接口，具体的实现类有NioEndpoint和Nio2Endpoint，有两个重要的子组件Acceptor和SocketProcessor
        1. Acceptor是用来监听Socket连接请求
        2. SocketProcessor用来处理接收到的Socket请求，它实现了Runnable接口，在run方法中调用协议处理组件Processor进行处理，SocketProcessor被提交到线程池中执行
  3. Processor
    1. 具体职责：Processor是用来实现HTTP协议，是对应应用层协议的抽象，Processor接受来自Endpoint的Socket，解析成Tomcat Request和Response对象，并通过Adapter提交到容器中
      1. Processor是个接口，定义了请求的处理等方法。它的抽象实现类 AbstractProcessor 对一些协议共有的属性进行封装，没有对方法进行实现。具体的实现有 AJPProcessor、HTTP11Processor 等，这些具体实现类实现了特定协议的解析方法和请求处理方式
  4. ProtocolHandler
    1. 通过组合实现: EndPoint 接收到 Socket 连接后，生成一个 SocketProcessor 任务提交到线程池去处理，SocketProcessor 的 Run 方法会调用 Processor 组件去解析应用层协议，Processor 通过解析生成 Request 对象后，会调用Adapter 的 Service 方法

Adapter

  1. 到了这一步，还需要进行转换ServletRequest
  2. ProtocolHandler接口的结果: 解析请求并生成 Tomcat Request 类。但是这个 Request 对象不是标准的 ServletRequest，也就意味着，不能用 Tomcat Request 作为参数来调用容器
  3. CoyoteAdapter:j这是适配器模式的经典运用，连接器调用 CoyoteAdapter 的 Sevice 方法，传入的是 Tomcat Request 对象，CoyoteAdapter 负责将 Tomcat Request 转成 ServletRequest，再调用容器的 Service 方法

最后再说下Tomcat的网络支持

  1. Tomcat 支持的 I/O 模型有
    1. NIO：非阻塞 I/O，采用 Java NIO 类库实现。
    2. NIO2：异步 I/O，采用 JDK 7 最新的 NIO2 类库实现。
    3. APR：采用 Apache 可移植运行库实现，是 C/C++ 编写的本地库。
  2. Tomcat 支持的应用层协议有
    1. HTTP/1.1：这是大部分 Web 应用采用的访问协议。
    2. AJP：用于和 Web 服务器集成（如 Apache）。
    3. HTTP/2：HTTP 2.0 大幅度的提升了 Web 性能。

2. Servlet容器

Servlet容器的结构

  1. Tomcat 设计了 4 种容器，分别是 Engine、Host、Context 和 Wrapper。这 4 种容器不是相互独立的关系，而是父子关系，逐层包含

  2. 容器间对应关系如下图所示

    1. 一个Service最多只能有一个Engine
    2. 一个Engine可以管理多台host（虚拟主机）
    3. 一个host下可以部署多部context，一个context表示一个web应用
    4. 一个context下可以有多个wrapper，一个wrapper就是一个servlet
    5. wrapper对servlet进行包装，管理生命周期等

  3. 这里的关系可以在tomcat中的server.xml配置文件中体现

  4. 补充一点设计思想吧

    1. 由于Tomcat中的这些容器是父子的包含关系，最后会形成一个树状结构，这就是设计模式中的组合模型，Tomcat就是通过组合模式来管理这些容器的

    2. 所有的容器组件都实现了Container接口，因此组合模式可以使得用户对但容器对象和组合溶剂对象的使用具有一致性。这里但容器对象是指最底层的Wrapper，组合容器对象指的是Wrapper之上的Context、Host或者Engine

    3. 给个Container的接口

      1. Container 接口扩展了 LifeCycle 接口，LifeCycle 接口用来统一管理各组件的生命周期

Servlet容器工作流程

  1. 定位工作: 在这么多个容器组合使用时，最重要的是确定请求要到达哪一个Servlet中进行处理
  2. Mapper组件
    1. Mapper里保存了Web应用的配置信息，就是容器组件与访问路径的映射关系，这里再用用老图
      1. Host容器里配置的域名
      2. Context容器里的Web应用路径
      3. 以及Wrapper容器里Servlet映射的路径
    2. S1: 根据协议和端口号选定 Service 和 Engine
      1. Tomcat 的每个连接器都监听不同的端口，而一个连接器是属于一个 Service 组件的，一个 Service 组件里除了有多个连接器，还有一个容器组件，具体来说就是一个 Engine 容器，因此 Service 确定了也就意味着 Engine 也确定了
    3. S2: 根据域名选定 Host
    4. S3: 根据 URL 路径找到 Context 组件
    5. S4: 根据 web.xml 找到 Wrapper（Servlet）
      1. Context 确定后，Mapper 再根据 web.xml 中配置的 Servlet 映射路径来找到具体的 Wrapper 和 Servlet

责任链模式实现定位原理

  1. 连接器先处理请求后，最先拿到请求的是 Engine 容器，Engine 容器对请求做一些处理后，会把请求传给自己子容器 Host 继续处理，依次类推，最后这个请求会传给 Wrapper 容器，Wrapper 会调用最终的 Servlet 来处理 –> 使用Pipeline-Valve管道来实现

  2. 什么是责任链?A: 类似链表：责任链模式是指在一个请求处理的过程中有很多处理者依次对请求进行处理，每个处理者负责做自己相应的处理，处理完之后将再调用下一个处理者继续处理

  3. Pipline-Value

    1. Valve接口如下，Valve 是一个处理点，因此 invoke 方法就是来处理请求的。 Valve 中有 getNext 和 setNext 方法 –> 类似于链表中的节点

    2. Pipeline接口如下，Pipeline 中有 addValve 方法。Pipeline 中维护了一个Valve链表，Valve 可以插入到 Pipeline 中，对请求做某些处理

      1. Pipeline 中没有 invoke 方法，因为整个调用链的触发是 Valve 来完成的，Valve 完成自己的处理后，调用 getNext.invoke() 来触发下一个 Valve 调用

  4. tomcat容器中的pipline

    1. Tomcat中每一个容器都有一个 Pipeline 对象
    2. 触发这个 Pipeline 的第一个 Valve，这个容器里 Pipeline 中的 Valve 就都会被调用到
    3. 这个 BasicValve 处于 Valve 链表的末端，它是 Pipeline 中必不可少的一个 Valve，负责调用下层容器的 Pipeline 里的第一个 Valve
    4. 不同容器的 Pipeline 通过getBasic 方法调用下层容器的Pipeline里的第一个Valve
    5. 由Adapter触发: 整个调用过程由连接器中的 Adapter 触发的，它会调用 Engine 的第一个 Valve

Valve和Filter?

  1. Valve 是 Tomcat 的私有机制，与 Tomcat 的基础架构 /API 是紧耦合的，它工作在容器级别，拦截所有应用的请求
  2. Filter 工作在应用级别，只能拦截某个 Web 应用的所有请求
  3. 如果想要一个整个web容器的filter，其实应该使用Valve来进行实现

连接器与Sevlet容器的架构作用

  1. Sevlet容器是一个路由与服务处理功能，负责内部处理
  2. 连接器就是用于封装网络结构的，负责外部交流
