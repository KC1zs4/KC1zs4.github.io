服务发现

  1. 始于需求：服务发现（Service Discovery）是微服务架构中的一个关键概念，在一个典型的微服务环境中，应用程序被分解成多个独立的服务，每个服务都运行在不同的进程中，甚至可能位于不同的物理或虚拟服务器上。这些服务需要相互通信来完成业务逻辑
    1. 存在问题：直接硬编码服务的地址和端口不仅难以维护，也限制了系统的灵活性和可扩展性
    2. 解决方案：设置服务注册中心来查找其他服务的位置，只需要寻找注册中心
      1. 服务注册：当一个新的服务实例启动时，它会向服务注册中心注册自己的位置信息（如主机名、IP地址和端口号）
      2. 服务查找：任何想要与其他服务通信的服务都可以询问服务注册中心以获取目标服务的位置信息。服务注册中心返回一个或多个可用的服务实例列表，调用方可以选择其中的一个进行通信
      3. 具有其他功能：服务健康（在线）检查，负载均衡，K8S自动缩放

RouteLocator

  1. RouteLocator

    1. 一个接口，用于定义和获取网关的路由规则，是请求转发和动态路由配置的关键a
    2. 定义路由：通过实现 RouteLocator 接口或使用其构建器（如 RouteLocatorBuilder），可以定义一系列的路由规则，这些规则决定了如何将传入的 HTTP 请求映射到不同的后端微服务
      1. 获取路由集合：RouteLocator 的主要方法是 getRoutes()，它返回一个Flux<Route>，即一个响应式的流，其中包含了所有已定义的路由对象。每个 Route 对象代表一条具体的路由规则。
      2. 动态更新：由于 RouteLocator 返回的是一个 Flux，可以支持异步和非阻塞的操作，并且能够处理动态变化的路由配置（例如，从外部源加载路由配置）

  2. RouteLocatorBuilder

    1. RouteLocatorBuilder是一个构建路由规则的帮助类
      1. builder.routes() 开始创建一组路由规则
      2. .route(“example-route”, …) 定义了一条名为 “example-route” 的路由
      3. .path(“/example/**”) 指定了该路由应匹配 /example/ 下的所有路径
      4. .filters(f -> f.addRequestHeader(…)) 添加了一个过滤器，用于在请求转发之前向请求头添加信息
      5. .uri(“http://example-service“) 指明了匹配请求的目标地址

  3. 路由顺序

    1. Spring Cloud Gateway中，路由的顺序按照它们在 routes() 方法中定义的顺序依次检查每个路由的匹配条件。一旦某个请求与某一路由规则相匹配，它就会被立即处理，并且不会再尝试匹配后续的路由规则。因此，路由定义的顺序会影响最终的路由行为
      1. 优先级：如果多个路由规则可能匹配同一个请求，则最先定义的那个会生效。
      2. 默认路由：通常建议将最具体或最常用的路由放在前面，而更通用的路由（如捕获所有未匹配路径的通配符路由）放在最后。
      3. 避免冲突：确保没有两个路由规则会对同一组请求产生歧义，除非想要这样的行为
    2. 最佳实践：先具体后广泛

Filter

这里着重关注于Spring Cloud的Filter

  1. for what: 和Spring的过滤器类似，cloud中的主要作用是在请求被转发到目标服务之前或响应返回给客户端之后对请求或响应进行修改、增强或处理

    1. 一句话身份验证、日志记录、重写路径、添加/修改请求头和响应头等功能(我爱ai)
    2. 预处理请求
      1. 身份验证和授权：确保只有经过验证的用户才能访问特定资源
      2. 请求头管理：添加、修改或删除请求头信息，例如添加 API 密钥、设置 CORS 头等
      3. 请求体转换：修改或转换请求体内容，适用于需要标准化输入格式的场景
      4. 路径重写：改变原始请求路径以适应后端服务的需求
    3. 后处理响应
      1. 响应头管理：添加、修改或删除响应头信息，如设置缓存策略、CORS 响应头等
      2. 响应体转换：修改或转换响应体内容，例如加密响应数据或压缩响应内容
      3. 错误处理：统一处理来自不同后端服务的错误响应，提供一致的错误反馈机制
    4. 监控和日志记录
      1. 性能监控：记录每个请求的处理时间，帮助分析系统性能瓶颈
      2. 访问日志：记录所有进出网关的流量，便于审计和故障排查
    5. 流量控制
      1. 限流：限制单位时间内某个 IP 或用户的请求数量，防止滥用或攻击
      2. 熔断降级：当后端服务不可用时，快速失败并返回友好的错误信息，避免影响其他正常的服务调用
    6. 路由选择优化
      1. 动态路由调整：基于请求特性（如地理位置、用户类型）动态选择最合适的后端服务
      2. 负载均衡：结合 Spring Cloud LoadBalancer 实现智能的请求分发，提高系统的可用性和响应速度

  2. 全局过滤器

    1. 应用于所有路由，通常用于实现跨切面的功能，如安全检查、日志记录等。它们在 GlobalFilter 接口中定义，并且可以在配置类中注册为 bean

  3. 路由过滤器

    1. 仅应用于特定路由规则中，通过 RouteLocatorBuilder 的 filters() 方法指定。它们更加细粒度地控制单个路由的行为，例如添加特定于该路由的请求头或执行路径重写
      1. 上面讲到的spring cloud中的情况

  4. 内置过滤器

    1. AddRequestHeader / AddResponseHeader：向请求或响应中添加头部信息
    2. RewritePath：根据正则表达式重写请求路径
    3. StripPrefix：移除请求路径中的前缀部分
    4. PreserveHostHeader：保持主机头不变，默认情况下会替换为主机的实际地址
    5. Hystrix：集成 Hystrix 熔断器来实现容错机制
    6. RateLimiter：基于令牌桶算法实现速率限制
    7. RequestRateLimiter：根据请求频率限制访问速率
