什么是Maven

  1. Maven 是一个项目管理工具，可以对 Java, C#，Ruby，Scala 和其他语言项目进行构建、依赖管理
  2. Maven是IDEA(用于java)自带的构建工具
  3. Maven的功能
    1. 构建
    2. 文档生成
    3. 报告
    4. 依赖
    5. SCMs
    6. 发布
    7. 分发
    8. 邮件列表

项目目录结构

  1. Maven项目的标准目录结构

    1. 有一个项目描述文件pom.xml
    2. 存放Java源码的目录是src/main/java
    3. 存放资源文件的目录是src/main/resources
    4. 存放测试源码的目录是src/test/java
    5. 存放测试资源的目录是src/test/resources
    6. 所有编译、打包生成的文件都放在target目录里

  2. 不要随意修改目录结构：所有的目录结构都是约定好的标准结构，IDEA会自动生成

    1. ~/.m2/repository: Maven默认的本地仓库目录位置
    2. ${basedir}/src/main/webapp/WEB-INF: web应用文件目录，web项目的信息，比如存放web.xml、本地图片、jsp视图页面

maven classpath

  1. Maven通过其构建生命周期自动管理项目的类路径(与maven build相关)
    1. 与常说的classpath区别与关系
      1. 基础：Maven Classpath 是 Classpath 的一种实现方式，专门用于Maven项目。Maven通过其插件机制和依赖管理系统，自动化了类路径的管理过程
      2. 目的：两者的目的都是为了确保Java应用程序能够找到并加载所需的类和资源文件，但在Maven项目中，这一过程由Maven自动完成
  2. 编译类路径（Compile Classpath）
    1. 项目源代码：位于src/main/java目录下的所有源代码
    2. 依赖库：从远程仓库下载的所有依赖项（包括直接依赖和传递依赖） –> scope: compile
    3. 资源文件：位于src/main/resources目录下的所有资源文件
  3. 运行类路径（Runtime Classpath）
    1. 编译后的类文件：位于target/classes目录下的所有.class文件
    2. 依赖库：从远程仓库下载的所有依赖项（与编译类路径类似，但不包括仅用于测试的依赖） –> scope: runtime, compile
    3. 资源文件：位于target/classes目录下的所有资源文件
  4. 测试类路径（Test Classpath）
    1. 测试源代码：位于src/test/java目录下的所有测试代码
    2. 测试资源文件：位于src/test/resources目录下的所有资源文件
    3. 编译后的类文件：位于target/classes目录下的所有.class文件
    4. 依赖库：从远程仓库下载的所有依赖项（包括直接依赖和传递依赖），以及测试范围的依赖项（如JUnit）
scope 编译类路径 (Compile Classpath) 测试类路径 (Test Classpath) 运行类路径 (Runtime Classpath)
compile 包含 包含 包含
provided 包含 包含 不包含
runtime 不包含 包含 包含
test 不包含 包含 不包含
system 包含 包含 不包含

pom.xml

pom必须结构

  1. POM( Project Object Model，项目对象模型 ) 是 Maven 工程的基本工作单元，是一个XML文件
  2. pom.xml基本结构和常用结构

    1. 必备结构

      1. 所有 POM 文件都需要 project 元素和三个必需字段：groupId，artifactId，version
      元素 描述
      project 项目的根元素，包含所有的子元素
      groupId 项目的组织标识符，通常是组织的域名倒序，比如：com.example
      artifactId 项目的唯一标识符，通常是项目名，比如：my-project
      version 项目的版本号，通常是 x.y.z 格式，比如：1.0.0

    2. IDEA自动生成的pom.xml

pom常用结构

  1. 常用结构
元素 描述
name 项目的名称
description 项目的描述
packaging 项目的打包方式，通常是 jar 或 war
properties 定义一些属性，比如编码、Java版本等
dependencies 定义项目的依赖库，包括 groupId、artifactId 和 version 三个元素
build 定义项目的构建配置，包括插件、资源等
profiles 定义项目的配置文件，用于不同环境的构建配置(特定的配置)
repositories 定义项目的仓库配置，用于下载依赖库

dependencies

  1. dependencies结构

    1. 在Maven中声明一个依赖项可以自动下载并导入classpath

    2. Maven使用groupId，artifactId和version唯一定位一个依赖

    3. 解释以下这里的<scope>: 这里是build阶段中的

      1. compile：默认值，适用于所有阶段，依赖项会被打包到最终的JAR或WAR文件中，确保应用程序在部署后能够正常运行 –>
        1. 项目的核心功能实现所必需的库。例如，Spring框架、Hibernate等
      2. provided：意味着该依赖将在编译和测试时使用，但在运行时由容器或JDK提供(环境提供) –>
        1. 使用场景：Servlet API、JSP API等，这些API通常由应用服务器提供
      3. runtime：表示该依赖不是编译所必需的，但需要在运行时和测试阶段存在 –>
        1. 数据库连接所需的驱动程序，如MySQL、PostgreSQL驱动
        2. 在运行时通过反射或其他机制动态加载的库
      4. test：仅在测试编译和执行阶段可用。这类依赖通常用于单元测试框架或测试工具 –>
        1. 单元测试框架：用于编写和运行单元测试的框架，如JUnit、TestNG
        2. 测试工具：辅助测试的工具，如Mockito、PowerMock
      5. import：仅限于<dependencyManagement>部分，用于从另一个POM中导入依赖管理配置(模块化)

packaging

  1. packaging

    1. jar：Java 应用程序
    2. war：Web 应用程序
    3. pom：父项目或聚合项目

properties

  1. properties

    1. 功能: 针对的主要是项目的一些配置属性

      1. 属性引用: 可以在 POM 文件的其他部分通过 ${property.name} 引用，支持跨模块复用

    2. 从Java 9开始，推荐使用maven.compiler.release属性，保证编译时输入的源码和编译输出版本一致。如果源码和输出版本不同，则应该分别设置maven.compiler.source和maven.compiler.target

      1. maven.compiler.release：表示使用的JDK版本，例如21；
      2. maven.compiler.source：表示Java编译器读取的源码版本；
      3. maven.compiler.target：表示Java编译器编译的Class版本。

build

  1. build
    1. 用于配置项目的构建过程，详见maven build

repositories

  1. repositories

    1. 如果仓库需要认证（如私有仓库），Maven 会使用 <id> 在 settings.xml 中查找对应的认证信息

    2. Maven 中央仓库的默认 <id>: central

    3. 对于私有仓库或其他第三方仓库，可以自定义 <id>

SNAPSHOT和RELEASE

  1. RELEASE：表示稳定版本，不会有变化 –> Release版本会被上传到远程仓库的发布库（release repository）中
    发布库中的文件一旦上传后不能被覆盖，确保了版本的唯一性和不可变性
  2. SNAPSHOT：表示开发中版本，不稳定，可能会有变化
    1. 自动更新
      1. Maven会自动检查远程仓库中是否有新的SNAPSHOT版本，并下载最新版本
      2. 这意味着每次构建时，可能会获取到不同的SNAPSHOT版本，因此构建结果可能不一致
    2. 快照存储
      1. SNAPSHOT版本会被上传到远程仓库的快照库（snapshot repository）中
      2. 快照库中的文件可以被覆盖，即同一个版本号的SNAPSHOT可以有多个不同的构建结果
    3. 时间戳版本
      1. 当上传SNAPSHOT版本时，Maven会在远程仓库中生成带有时间戳的文件名，例如 my-app-1.0-20250214.100500-1.jar

maven build

build是Maven的核心功能之一，用于构建项目，包括编译、测试、打包、部署等

生命周期

  1. build生命周期

    1. 为了完成 default 生命周期，以下阶段（包括其他未在上面罗列的生命周期阶段）将被按顺序地执行 –> 一般到verify就o了
    2. 在一个生命周期中，运行某个阶段的时候，它之前的所有阶段都会被运行

    阶段 描述
    validate 验证项目是否正确且所有必须信息是可用的
    compile 编译源代码
    test 使用适当的单元测试框架（例如JUnit）运行测试
    package 将编译后的代码打包成可分发的格式，例如 JAR和WAR
    verify 对集成测试的结果进行检查，以保证质量达标
    install 安装打包的项目到本地仓库，以供其他项目使用
    deploy 拷贝最终的工程包到远程仓库中，以共享给其他开发人员和工程

  2. Maven有3个标准生命周期

    1. clean生命周期: 清理项目，删除目标目录中的编译输出文件(out)。这通常是在构建之前执行的，以确保项目从一个干净的状态开始
    2. Default生命周期(项目):（也称为 Build 生命周期）
      1. validate
      2. compile
      3. test
      4. package
      5. verify
      6. install
      7. deploy
    3. Site生命周期(信息): 生成项目站点文档，发布站点文档
      1. site：生成项目文档和站点信息
      2. deploy-site：将生成的站点信息发布到远程服务器，以便共享项目文档

  3. 构建阶段由插件目标构成: 略

配置文件:profile

profile介绍与结构

  1. 构建配置文件是一系列的配置项的值，可以用来设置或者覆盖 Maven 构建默认值: 比如properties、dependencies、build等

    1. 不同的环境：生产环境（Production），开发（Development）环境定制构建方式

    2. pom.xml中指定: 在 pom.xml 文件中使用 activeProfiles 或者 profiles 元素指定配置文件/配置内容!!!

      1. 配置文件在构建时修改 POM(模型)，并且用来给参数设定不同的目标环境（比如开发（Development）、测试（Testing）和生产环境（Production）中数据库服务器的地址）

    3. 三种类型

      类型 描述
      项目级(Per Project) 指定在项目的POM文件pom.xml中(./pom.xml)
      用户级(Per User) 指定在Maven的设置xml文件中 (%USER_HOME%/.m2/settings.xml)
      全局(Global) 指定在 Maven 全局的设置 xml 文件中 (%M2_HOME%/conf/settings.xml)

    4. 激活方式(一般用前3种)

      1. 使用命令控制台输入显式激活
      2. 通过 maven 设置
      3. 基于环境变量（用户或者系统变量）
      4. 操作系统设置（比如说，Windows系列）
      5. 文件的存在或者缺失

  2. 基本结构

    标签 描述
    <id> Profile 的唯一标识符，用于在命令行中激活或引用。
    <activation> 定义 Profile 的激活条件（可选）。
    <properties> 定义自定义属性，可以在构建过程中使用。
    <dependencies> 定义该 Profile 特有的依赖。
    <build> 定义该 Profile 特有的构建配置（如插件、资源等）。

Maven换源

mirrors and repositories

  1. 源(mirrors)与repositories

    1. repositories

      1. 含义：Maven仓库是存放和获取构件（如JAR文件、POM文件等）的地方。这些仓库可以是本地的，也可以是远程的
      2. 分类
        1. 本地仓库（Local Repository）：位于开发者本地机器上的目录，默认路径为 ~/.m2/repository。当Maven从远程仓库下载依赖时，会将这些依赖存储在本地仓库中以便后续使用
        2. 中央仓库（Central Repository）：由Maven社区维护的一个公共仓库，包含了大量的开源库
        3. 远程仓库（Remote Repository）：除了中央仓库外，还可以配置其他远程仓库，通常是私有仓库或第三方提供的仓库
      3. 作用
        1. 依赖管理：通过指定多个远程仓库，确保Maven能够找到并下载所需的依赖项
        2. 发布构件：开发者可以通过配置将自己项目的构件发布到远程仓库中，供其他项目使用

    2. Mirrors

      1. 含义：Maven镜像是对某个仓库的完整副本，通常用于提高下载速度或增强可靠性。镜像服务器会定期同步源仓库的内容，以保持数据的一致性
      2. <mirrorOf>: 指定了哪些仓库（repositories）会被镜像服务器替代

      2. 用户配置：位于用户目录下的 .m2/ 文件夹中（如：~/.m2/settings.xml）

一些实践与解决方案

mvn wrapper

mvnw介绍与结构

  1. 什么是mvn wrapper

    1. Maven Wrapper（简称 mvnw）是一个用于在没有预先安装 Maven 的环境中运行 Maven 项目的工具。它允许开发者将 Maven 的二进制文件和配置文件与项目一起打包，从而确保**所有开发者或构建环境使用相同版本的 Maven (跨平台)**，避免因 Maven 版本不一致导致的问题

  2. Maven Wrapper 包含以下文件

    1. mvnw：Unix/Linux/MacOS 下的可执行脚本
    2. mvnw.cmd：Windows 下的可执行脚本
    3. .mvn/wrapper/ 目录
      1. maven-wrapper.jar：Maven Wrapper 的核心实现。
      2. maven-wrapper.properties：指定要使用的 Maven 版本
      3. 当运行 mvnw 时，它会检查本地是否已经下载了指定版本的 Maven。如果没有，它会自动从 Maven 官方仓库下载并缓存到用户目录下的 .m2/wrapper/ 中

为项目添加mvnw

  1. CASE1: 使用maven插件生成

    1. Maven 提供了 maven-wrapper 插件，可以快速为项目生成 Maven Wrapper

maven模块化

父子pom

  1. 父POM配置

    1. packaging：父模块的packaging必须设置为pom
    2. 模块声明：在<modules>中列出所有子模块
    3. 依赖管理：使用<dependencyManagement>统一管理依赖版本，子模块需显式声明依赖
    4. 插件管理：使用<pluginManagement>统一管理插件配置

  2. 子POM配置

    1. 父模块引用：子模块需通过<parent>引用父模块
    2. 依赖声明：子模块需在<dependencies>中声明依赖，版本由父模块管理。
    3. 独立性：子模块应尽量独立，避免循环依赖

模块化管理

  • 模块化管理好处
    • 统一版本：在父模块中统一管理依赖版本，子模块无需重复指定
    • 避免冲突：确保依赖版本一致，避免冲突

  1. 依赖管理

    1. 核心思想：父POM（Project Object Model）文件可以集中管理依赖库的版本号，子模块中只需要引用依赖不需要再指定版本号，确保所有子模块都使用相同的版本，并且简化了依赖的升级过程

    2. 父pom.xml：使用<dependencyManagement>元素来声明所有需要管理的依赖项。在这里，你应该指定依赖的groupId、artifactId和version。这些信息将用于定义依赖库的标准版本

    3. 子pom.xml：子模块的pom.xml应该继承自父POM。在子模块中，你只需要引用依赖(自动引用)而不需要再次声明版本号，因为版本已经被父POM中的<dependencyManagement>部分所管理

  2. 插件管理

    1. 统一配置：在父模块中统一配置插件，子模块可继承或覆盖
    2. 统一管理插件版本，也可以使用<pluginManagement>元素在父POM中声明插件及其版本，然后在子模块中仅引用插件而不必重复版本信息
